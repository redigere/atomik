# Atomik Rust Migration Sprint Plan

This document defines the complete migration roadmap for transitioning the Atomik operating system configuration runtime from Python 3 into a native Rust binary. The migration is organized across ten sprints, each targeting a specific architectural layer. Every sprint produces a fully compilable, production-ready artifact with zero fallback, zero placeholder stubs, and zero deferred work items. The final deliverable is a self-contained Rust CLI that matches the Python implementation in every observable behavior while delivering measurably superior startup latency, memory footprint, and error diagnostics.

## Sprint Zero Project Scaffold and Cargo Foundation

The first sprint establishes the entire Rust project skeleton, build toolchain, and dependency graph. A Cargo.toml is created at the repository root declaring all external crates that the final binary depends upon. The module tree mirrors the Python source layout exactly with source files under a src directory organized into config, lib, and tasks subdirectories. Each module file receives its mod declaration and a minimal type or function signature that establishes the compilation boundary without any implementation logic. The workspace compiles cleanly with cargo check at the end of this sprint, proving that every module path resolves correctly and every external crate interface is properly imported. The Rust edition is set to 2024, and the output binary is named atomik to match the existing CLI entry point.

## Sprint One Configuration Loading and Dot Notation Access

The second sprint implements the core configuration subsystem that reads YAML declaration files from the config directory. A Config struct is defined that loads a named YAML file through serde yaml and provides the get method returning an optional reference for safe access, the require method that returns an error on missing keys, and the navigate internal method that splits dot separated paths and traverses the nested serde yaml Value tree. The interpolate method accepts a format string with brace placeholders and a HashMap of substitution values, uses the standard library format macro with named arguments, and splits the result into a command vector using shlex compatible logic. The expand commands method loads command arrays from config sections and applies interpolation across every entry. The validate method accepts a schema dictionary mapping key paths to expected types and produces a list of validation error strings. The module also provides typed accessor structs for the pipeline configuration that deserialize the steps dictionary into a strongly typed enum, enabling the pipeline orchestration layer to match on step names without string comparison. All config error messages use consistent formatting with the config file name prefix to aid debugging.

## Sprint Two Shared Utilities and Command Execution

The third sprint delivers the shared execution layer that all task modules depend upon. The core function is execute shell command which converts command representation strings into argument vectors, optionally prefixes them with sudo, applies configurable timeouts, captures stdout and stderr, and exits the process fatally on nonzero return codes unless the fatal flag is explicitly set to false. The dry run mode prints commands without executing them. The sudo user prefix construction queries the commands configuration for the correct prefix template and applies format substitution with the username and D Bus address. The dconf writing subsystem implements write dconf setting with a three second timeout from configuration, write gsettings setting with its own configurable timeout, and the full batch write protocol that creates a complete backup, iterates over setting key value pairs, and triggers automatic restore on any single failure. The RPM package query functions parse rpm qa output and ostree status JSON to determine package installation state, pending reboots, staged deployments, and system unlock status. The desktop environment availability check verifies that both the dconf binary is installed and a D Bus session is reachable before allowing dconf operations. All timing parameters are loaded from the commands YAML configuration rather than being hardcoded.

## Sprint Three Variant Detection and Progress Tracking

The fourth sprint implements the runtime variant detection and user facing progress reporting subsystems. The VariantContext struct reads the variants configuration and iterates through each variant definition, checking binary existence with the which utility and ostree refspec containment through JSON parsing of rpm ostree status output. The first matching variant becomes the active context and determines the pipeline sequence through the pipeline method, the desktop environment type through the desktop method, and variant specific properties through the general property accessor. Boolean convenience methods for is gnome, is kde, and is cosmic allow task modules to branch conditionally. The ProgressTracker struct maintains a step label list, a completed count, and a current step position. The configure method resets all counters for a new pipeline run. The start step method logs the percentage completion and step label using the square bracket format required by project conventions. The advance within step method computes fractional progress within the current step. The complete step method increments the completed count and logs the completion percentage. The complete method emits the final all steps complete message. Both the variant context and progress tracker are instantiated as global singleton statics using the once cell or lazy lock pattern for thread safe lazy initialization.

## Sprint Four Pipeline Orchestration and Transaction Management

The fifth sprint delivers the pipeline engine that sequences task execution with transactional guarantees. The pipeline configuration is loaded at startup and parsed into a sequence of step descriptors, each containing a step name, a requires reboot flag, and a boxed closure or function pointer to the task entry point. The pipeline functions are registered at compile time through a match expression rather than through runtime dynamic imports. The build pipeline function constructs the ordered step list by consulting the variant context pipeline method and falling back to the default sequence. The execute requested configuration steps function iterates over the ordered steps, handles reboot pending detection with force mode override, captures the system unlock state, prompts for user confirmation unless dry run or assume yes mode is active, and wraps the entire sequence in a transaction boundary. The TransactionManager struct creates a timestamped temporary directory at the begin call and captures the booted deployment identifier. The snapshot step method dispatches on step name to capture RPM package lists, security configuration files, KDE config files, third-party repository files, extension directories, font lists, and container configuration files. The rollback method reverses the completed steps in inverse order, restoring files from backups, removing configured third-party repositories, deleting newly installed extensions, and cancelling staged package operations. The commit method removes the backup directory and transitions the transaction to the completed state. Step failure at any point triggers immediate rollback and a fatal error message.

## Sprint Five Package Management and RPM Ostree Operations

The sixth sprint implements the packages module that manages RPM ostree package installation and removal. The install packages function queries the variant specific remove list and the base packages configuration install list. It computes the set of packages that need removal by comparing installed packages against the remove list while excluding already removed packages. It computes the set of missing install packages by querying RPM for each required package. Removal operations use the rpm ostree override remove command with the full package list in a single invocation. Installation operations iterate individually and call rpm ostree install y idempotent. After all removals and installations complete, the function invokes rpm ostree apply live allow replacement to activate the changes without a reboot and marks the transaction as having applied live changes. The progress tracker advance within step method is called after each individual operation with a fraction reflecting the total operation count. The force mode flag causes the function to cancel any pending deployment and clean up cached state before proceeding.

## Sprint Six Security Hardening and System Protection

The seventh sprint implements the comprehensive security hardening module that configures firewalld, auditd, fail2ban, SSH, sysctl parameters, sudo, file permissions, resource limits, systemd oomd, earlyoom, OOM protected service drop ins, and desktop compositor scheduler drop ins. The apply security hardening function loads all configuration sections from the security YAML file and the commands YAML file. Firewall configuration enables the firewalld service, iterates over service specific commands for each declared service, and reloads the firewall. Audit configuration enables the auditd service and writes the audit rules file from the configuration section list. Fail2ban configuration enables the service and writes the jail configuration. SSH configuration enables sshd and writes the SSH daemon configuration drop in. Sysctl parameters are flattened from the nested dictionary structure and each parameter is applied both through the sysctl w command and written to the persistent configuration file. Sudo configuration is written with strict file permissions loaded from configuration. Resource limits are formatted from the limits dictionary into the standard limits conf file format. Systemd oomd configuration enables the service and writes its configuration drop in and user slice drop in. OOM protected service drop ins are created for each variant specific service name using the configured drop in filename. Desktop compositor scheduler drop ins apply FIFO scheduling policy with priority 45 for the variant specific compositor services. Earlyoom configuration writes the environment file, the systemd unit override, enables the service, and starts it. The progress tracker advance within step method is called after every individual operation with a fraction calculated from the pre computed total operation count.

## Sprint Seven Desktop Settings and Asset Management

The eighth sprint delivers the desktop settings module and the assets module as a combined milestone. The apply desktop settings function is gated behind GNOME variant detection and GDM health verification with a three second timeout. Dconf settings are built from the configuration dictionary with enabled extensions substitution, and written through the batch write protocol with automatic backup and rollback. Gsettings are applied individually with error logging but non fatal continuation. The GTK4 theme is copied from the system Adwaita theme directory to the user config directory. The default text editor association is set through xdg mime. The App Nap service script and systemd unit are written to their respective paths with correct permissions, followed by daemon reload and service enablement. The assets module removes unused system fonts through RPM ostree override remove, downloads Adwaita font archives from the canonical URLs, extracts TrueType font files into the destination directory, and refreshes the font cache. The wallpaper subsystem removes default GNOME background packages, downloads Adwaita wallpaper archives, filters images by keyword, and generates GNOME background properties XML on GNOME variants. The local wallpaper file from the assets directory is copied to the user backgrounds directory with correct ownership and permissions, and the GSettings wallpaper URI is updated on GNOME variants.

## Sprint Eight Extension Management and Repositories Configuration

The ninth sprint implements the GNOME Shell extension subsystem and the third-party repository subsystem. The install required ego extensions function runs only on GNOME variants. It loads the extension name list and its UUID label mapping from configuration, iterates through each extension, queries the extensions gnome org API for the current shell version, downloads the compatible extension ZIP archive, installs it through the gnome extensions command, and enables it. The function then enables all configured extensions, removes unauthorized user extensions that are not in the allowed list, and restarts the GNOME Shell through D Bus. The enable all configured extensions helper issues enable commands for every UUID in the allowed list without failure sensitivity. The remove unauthorized user extensions function scans the user extensions directory and deletes any directory whose UUID is not in the allowed list. The repos module downloads external .repo files and imports GPG keys for third-party native applications.

## Sprint Nine KDE Plasma and Toolbox Configuration

The tenth sprint completes the task module suite with KDE Plasma settings and Toolbox container setup. The apply kde settings function runs only on KDE variants. It writes theme, color scheme, font, monospace font, icon theme, and cursor theme values through the kwriteconfig5 command using the kde config and kde config group command templates. The taskbar layout plugin is set to icon tasks with fill width enabled. The KDE wallet is disabled through its configuration file. Desktop animations are disabled by setting animation duration factor to zero and disabling compositor blur and OpenGL unsafe mode. Power management is configured with button events set to sleep and a five minute idle timeout. The containers module creates Toolbox containers with development tooling to provide isolated developer environments.

## Sprint Ten System Reset, Backup, and Compliance Checking

The eleventh sprint delivers the system reset flow, the settings backup utility, and the compliance checking subsystem. The main reset flow prompts for confirmation, resets dconf on GNOME variants, removes the desktop wallpaper, deletes user installed GNOME extensions, removes Adwaita fonts, disables and removes the App Nap service, uninstalls all configured RPM packages, resets overrides for all removed packages, and removes configured third-party repositories. The create system settings backup function runs on GNOME variants, creates the backup directory under the user home, executes a complete dconf dump, and writes the output to the configured backup filename. The compliance checking subsystem provides functions for finding missing RPM packages, detecting installed unused fonts, verifying dconf setting values against expected values, finding unauthorized GNOME extensions, and running the full system scan. Each function returns a vector of issue descriptors that the CLI presents to the user grouped by issue category. The CLI dispatches the check command to execute the full system scan and the status command to run a single step check with the detected variant context.

## Sprint Eleven Performance Regression Testing and CI Profiling

The twelfth sprint establishes strict performance budgets to guarantee that the Rust rewrite delivers absolute maximum speed and minimal memory footprint. A dedicated performance testing suite is added to the CI pipeline to track binary execution time, memory allocations, and parsing latency for the configuration modules. The suite enforces hard thresholds: any commit that introduces an execution delay or memory bloat beyond the defined baseline will fail the pipeline automatically. This sprint guarantees that Atomik continues to operate instantly on any device and prevents silent regressions that could compromise the responsiveness of the configuration lifecycle.

## Sprint Twelve Atomic System Rollback and Snapshot Integration

The final sprint integrates deep OS-level safety mechanisms by leveraging the immutable properties of Fedora Atomic desktops. Instead of relying solely on the application's step-by-step reversal routines, this phase introduces native system snapshotting. Before any pipeline execution that modifies system state, Atomik will explicitly trigger an rpm-ostree deployment pin or native Btrfs snapshot. If an irrecoverable error or system lockup occurs during the application of configurations or daemon installations, the user or the CLI itself can invoke a direct rollback to the exact pre-execution state. This achieves true zero-risk execution, bringing server-grade atomic reliability directly to the desktop administration flow.
